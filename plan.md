# SubnetDesk LAN-only 单用户远程桌面改造方案

## 1. 文档目的

本方案用于将当前 RustDesk 改造为一个仅在局域网或已建立的 VPN 网络中运行的远程桌面工具。最终用户使用方式类似微软远程桌面：

1. 被控端安装并启动本机服务。
2. 被控端设置唯一一组用户名和密码。
3. 控制端输入 `IP/域名 + 端口 + 用户名 + 密码`。
4. 通过直接 TCP 连接进入远程会话。

改造后不再保留 RustDesk ID、公共/私有 rendezvous server、NAT 打洞、relay 中继、云端账号或跨网管理能力，但必须保留已有远程会话功能。

---

## 2. 已确定的产品决策

### 2.1 唯一连接模式

- 产品只有 `LAN-only` 运行模式，最终代码中不保留可切换回传统 RustDesk 公网模式的设置。
- 控制端只接受以下目标格式：
  - `192.168.1.20`
  - `192.168.1.20:21118`
  - `host.lan:21118`
  - `[fd00::20]:21118`
- 未显式输入端口时使用固定默认端口 `21118`。
- VPN 不是新的协议模式。WireGuard、Tailscale、OpenVPN 等只负责提供可路由 IP，应用层仍使用同一条 LAN 直连链路。

### 2.2 单用户认证

- 每台被控设备只保存一组 `access_username + access_password`。
- 这是应用自己的访问凭据，不是 Windows/macOS/Linux 系统账号。
- 不实现用户列表、角色、权限组、账号过期、组织或远程管理。
- 用户名和密码同时匹配时才允许建立会话。
- 认证成功后默认获得全部已保留远程功能，不再进行应用层授权判断。
- “单用户”表示只有一组凭据，不表示只能建立一个 TCP 连接。桌面、文件传输、终端、端口转发等现有并行通道以及多控制端并发能力继续保留；它们共享同一组凭据和同一套全功能会话语义。

### 2.3 “不需要权限控制”的准确含义

需要删除或绕过的是：

- 按用户/角色区分的远程功能权限。
- 连接管理端对每个会话动态开放键盘、剪贴板、音频、文件等功能的授权流程。
- `ControlPermissions` 或类似的服务端下发权限位图。
- “点击接受”作为密码认证之后的第二道人工授权。

仍然必须保留的是：

- macOS 屏幕录制/辅助功能、Android 无障碍、Linux Wayland portal 等操作系统必需权限。
- “仅查看”、“禁用本次会话音频”等控制端会话偏好；它们是功能开关，不是服务端授权。
- 平台不具备某项能力时的 capability 检测，例如不支持隐私模式的平台不能强制开启该功能。

### 2.4 协议兼容策略

- LAN-only 新客户端和新被控端必须配套使用。
- 不要为旧 RustDesk 客户端保留明文、旧设备密码或 rendezvous 兼容分支。
- 修改登录握手协议时提升协议版本，版本不匹配时给出明确错误并断开，不做降级尝试。

## 3. 范围边界

### 3.1 必须保留的远程功能

下列功能的业务实现和 UI 入口均不应因删除公网连接链路而丢失：

- 远程屏幕查看和键鼠控制。
- 多显示器切换、分辨率、缩放、画质和编解码设置。
- 远程音频播放。
- 文本剪贴板。
- 文件剪贴板和独立文件管理/文件传输。
- 聊天和语音通话。
- 查看摄像头。
- 端口转发。
- RDP tunnel 功能。
- 远程终端。
- 远程重启。
- 会话录制。
- 隐私模式。
- 锁定被控端本地输入。
- 远程打印。
- 白板/远程标注。
- 虚拟显示器、显示器插拔和显示模式切换。
- 远程锁屏及其他现有会话系统动作。
- 重连、多窗口/多标签和近期连接。
- 已有的跨平台屏幕采集、输入注入和编解码能力。
- LAN 设备发现可作为输入 IP 之外的辅助功能保留，但不能成为建立连接的必要依赖。

### 3.2 必须从最终产品移除的能力

- RustDesk 设备 ID 生成、展示、查找和连接。
- rendezvous server 注册、心跳、公钥签发和 ID 解析。
- 公共服务器和自定义 ID server 配置。
- relay server 配置、选择、回退、提示和状态展示。
- TCP/UDP/IPv6 NAT 检测和打洞。
- 强制 relay 与“直连/中继”切换。
- 服务器账号登录、组织、群组和云端地址簿。
- 云端在线状态、审计 API、部署注册和跨服务器功能。
- 面向公网服务的 key、token、API server 和 proxy 配置页。
- 后台公网自动更新检查；LAN-only 发行版通过离线安装包、内部分发或操作系统包管理方式升级，不在运行时依赖公网更新服务。

### 3.3 不应在本次顺手重写的部分

- 屏幕捕获、编解码器、音频编解码。
- 键盘鼠标输入注入。
- 剪贴板、文件传输、摄像头、终端的业务协议。
- Flutter 远程画布和远程工具栏。
- 各平台服务安装和提权机制，除非它们直接依赖被删除的公网链路。

## 4. 目标用户流程

### 4.1 被控端首次设置

1. 用户安装应用和被控端系统服务。
2. 应用检查屏幕录制、输入控制等操作系统权限。
3. 用户设置唯一的访问用户名和密码。
4. 用户选择监听网卡/地址，或接受默认 LAN/VPN 网卡。
5. 服务启动后显示：
   - 当前可连接 IP 列表。
   - 监听端口。
   - 设备名称。
   - 设备公钥指纹。
   - 服务运行状态。
6. 当用户名、密码或监听配置未完成时，服务不对外接受远程会话。

### 4.2 控制端连接

1. 用户输入 IP/域名、用户名和密码。
2. 客户端校验地址和端口格式。
3. 客户端直接建立 TCP 连接，不执行任何服务器查询或 NAT 测试。
4. 双方完成设备身份和会话加密握手。
5. 首次连接时显示被控端指纹，用户确认后按设备指纹建立信任记录。
6. 在已加密通道中发送单用户凭据并验证。
7. 认证成功后直接进入远程桌面，不弹出被控端人工接受对话框。
8. 后续文件传输、摄像头、终端、端口转发等会话类型复用同一认证和加密入口。

### 4.3 重连和凭据记忆

- 近期连接以“设备指纹”为稳定标识，IP 只是可变的最后地址。
- 默认可记住 IP、端口和用户名。
- 如保留“记住密码”，密码或可直接重放的验证材料必须进入操作系统凭据库，不能明文或仅 Base64 保存在 `PeerConfig`。
- 被控端指纹发生变化时必须停止自动重连并要求用户重新确认。

## 5. 目标架构

```text
控制端 Flutter
  └─ IP/域名 + 端口 + 用户名 + 密码
       └─ EndpointParser
            └─ LanConnector
                 └─ 设备指纹校验 + 强制加密握手
                      └─ SingleCredentialAuth
                           └─ 现有 Connection / io_loop
                                ├─ 桌面视频
                                ├─ 键鼠输入
                                ├─ 音频/剪贴板
                                ├─ 文件/摄像头/终端
                                └─ 端口转发和其他已有会话功能

被控端系统服务
  └─ LanServer
       ├─ 监听网卡/地址和端口
       ├─ 来源网段检查
       ├─ 设备长期密钥
       ├─ 加密握手
       ├─ 单用户认证和失败退避
       └─ 现有 Server / Connection 服务集
```

架构核心原则：只替换“怎样找到设备、建立连接和完成认证”，不重写“连接后如何提供远程功能”。

## 6. 详细技术设计

### 6.1 拆分直连监听器

当前直连监听器位于 `src/rendezvous_mediator.rs`，并由 `RendezvousMediator::start_all()` 顺带启动。改造后应：

- 新建独立的 `src/lan_server.rs` 或等价模块。
- 将 `get_direct_port()`、`direct_server()` 及其生命周期迁入该模块。
- 由 `src/server.rs` 或系统服务主入口直接启动 `LanServer`。
- 不再调用 `RendezvousMediator::start_all()`。
- 不再启动 NAT 类型测试、`hbbs_http::sync` 或任何注册任务。
- 保留 `src/lan.rs` 的 LAN 发现能力，但将广播内容改为设备名、IP、端口和公钥指纹，不再广播 RustDesk ID。

`LanServer` 必须支持：

- IPv4 和 IPv6。
- 固定默认端口与自定义端口。
- 指定监听地址，而不是只能 `listen_any()`。
- 多个网卡/地址。
- 可配置允许网段，以兼容 RFC1918、IPv6 ULA、Tailscale CGNAT 和显式配置的 VPN 网段。
- 停止服务或修改监听配置后可正常关闭并重新绑定，不遗留监听端口。

### 6.2 简化客户端建连链路

当前 `src/client.rs` 已对 IP 和 `domain:port` 有直连快速分支，但整个 `Client::_start()` 仍包含 ID 解析、rendezvous、NAT 和 relay 路径。改造后应：

- 将端点解析抽成单一 `Endpoint` 结构，包含 `host`、`port` 和原始显示值。
- 将建连入口收敛为 `LanConnector::connect(endpoint)`。
- 删除非 IP/域名分支，不再查询 `get_rendezvous_server()`。
- 删除 UDP 打洞、IPv6 打洞、relay request 和 force-relay 逻辑。
- 连接失败时直接返回可操作错误：DNS 失败、路由不可达、连接被拒绝、超时或协议不兼容。
- 成功连接后继续使用现有 `client/io_loop.rs` 处理远程功能消息。

连接地址解析必须对端口范围、IPv6 括号、过长域名、控制字符和空白做明确校验，不能把未验证输入插入 shell 命令或日志上下文。

### 6.3 设备身份和强制加密

现有 `secure_connection()` 依赖 rendezvous server 签发的 `signed_id_pk`。纯 IP 直连时拿不到该数据，当前代码可以继续使用未加密连接。LAN-only 模式不能继承这个行为。

目标握手：

1. 被控端首次启动时生成并持久化设备长期签名密钥。
2. 每次连接生成临时密钥，并由长期设备密钥签名握手。
3. 客户端完成指纹确认或校验已保存的设备公钥。
4. 双方建立带完整性保护和防重放的加密会话。
5. 加密或身份校验失败立即断开。

必须删除以下行为：

- 缺失签名公钥时发送空消息并返回未加密流。
- 公钥不匹配时 `fall back to non-secure`。
- 因目标是 IP 而跳过不安全连接检查。
- 把 VPN 链路本身当成应用层身份校验。

握手优先复用项目已有 libsodium 密钥和 `secretbox` 数据通道，但不能为了避免修改协议而保留任何明文降级。

### 6.4 单用户凭据存储

被控端配置模型只需要：

```text
access_username
access_password_hash
access_password_salt / PHC encoded parameters
credential_revision
```

约束：

- 用户名保存前去除首尾空白，设置最大长度并禁止控制字符。
- 密码不得明文写入配置文件、日志、panic 信息或审计事件。
- 密码存储使用 Argon2id 等面向密码的内存硬哈希，参数由实际目标设备基准确定。
- 配置文件和设备私钥文件必须使用现有系统服务 ACL/文件权限机制保护。
- 更改用户名或密码时递增 `credential_revision`，并立即使已缓存的认证会话失效。
- 客户端和服务端处理明文密码的临时内存应限制在登录阶段，并在验证结束后主动清理；不得把明文密码复制进长期会话结构。
- 首次升级时不默默把旧“永久密码”当成新凭据；显示一次设置流程，要求用户设置新用户名和密码。

### 6.5 登录协议

当前 `LoginRequest.username` 传输的是目标 ID/IP，不是认证用户名；`OSLogin` 又承载了系统会话凭据。不能在新产品中继续混用这些字段。

建议将协议整理为明确语义：

```text
LanClientHello
  protocol_version
  client_nonce
  client_capabilities

LanServerHello
  protocol_version
  server_nonce
  device_public_key
  ephemeral_public_key
  signature

LanLoginRequest       # 仅在加密通道内
  access_username
  access_password
  credential_revision_hint
  connection_type
  option

LanLoginResponse
  success | generic_error | retry_after
  peer_info
```

实际 protobuf 可根据现有 `LoginRequest` 的兼容成本选择新消息或新字段，但必须满足：

- IP/域名是客户端端点，不再作为服务端登录用户名。
- 应用访问用户名与 `OSLogin` 完全分离。
- 服务端不再校验请求中的 ID 是否等于 `Config::get_id()`。
- 终端或 Linux headless 如仍需内部 OS 会话信息，必须作为远程功能参数独立处理，不能代替 LAN 访问认证。
- 同一组凭据允许建立多个不同 `connection_type` 的并行通道；服务端仍需为每个连接维护独立的会话状态、超时和资源清理。

认证执行必须放在已完成设备身份校验的加密通道内。服务端使用 Argon2id 校验密码，不将可直接重放的密码哈希当作长期会话凭据。

### 6.6 登录失败保护

即使只在 LAN/VPN 中使用，也必须保留暴力破解防护：

- 连续失败后按来源 IP 做递增退避。
- 另设全局认证并发上限，避免多 IP 并发拖垮 Argon2 验证。
- IPv6 按地址和前缀同时限制，避免通过快速更换 IPv6 地址绕过。
- 客户端统一显示“用户名或密码错误”，不泄露用户名是否正确。
- 限制消息必须返回可控的 `retry_after`，客户端不能在倒计时内自动重试。
- 成功登录只清理对应来源的失败状态，不应无条件清除全局防护。

应尽量复用 `src/server/login_failure_check.rs` 现有的 IPv4/IPv6 失败检测、退避和并发 gate，但需要将 scope 改为新的单用户 LAN 认证语义。

### 6.7 认证后的功能处理

- 单用户认证成功后将连接标记为 `authorized = true`。
- 文件传输、摄像头、端口转发、终端和默认桌面会话共享同一登录过程。
- 删除 `ControlPermissions` 对 LAN 会话功能的决策作用。
- 删除未授权会话进入 Connection Manager 等待手动点击接受的流程。
- 不删除会话类型自身的平台可用性检查。
- 用户在控制端主动选择“仅查看”或关闭剪贴板时，继续通过现有 `OptionMessage` 表达本次会话偏好。

### 6.8 配置模型

最终产品保留的核心网络和认证配置：

```text
lan-listen-addresses
lan-listen-port
lan-allowed-networks
lan-discovery-enabled
access-username
access-password-hash
credential-revision
device-keypair
trusted-device-fingerprints       # 控制端本地配置
recent-endpoints                  # 控制端本地配置
```

应从生产运行路径和设置页移除：

```text
id-server / rendezvous-server
relay-server
api-server
key
custom-server
force-always-relay
udp-hole-punching
nat-type
access-token
server-account / group / address-book cloud state
```

对旧配置采用一次性迁移：

- 识别首次启动 LAN-only 版本。
- 忽略公共/私有服务器配置，不发起任何连接。
- 保留画质、显示器、剪贴板、编解码、语言等与会话功能相关的本地配置。
- 引导用户设置新单用户凭据和监听网卡。
- 迁移完成后记录本地 schema 版本，不在每次启动重复迁移。

## 7. Flutter 产品界面改造

### 7.1 主页

将当前“Remote ID”连接卡片改为：

- 计算机：IP、域名或 `IP:port`。
- 用户名。
- 密码。
- 记住密码（如实现安全凭据库）。
- 连接。
- 保留文件传输、摄像头、终端等已有连接类型入口。

主页不再显示：

- 本机 RustDesk ID。
- 临时密码、永久设备密码与新单用户凭据并列的旧入口。
- 公共服务器在线状态。
- 账号登录、组织和云地址簿。
- relay 或“直连/中继”状态。

### 7.2 被控端设置页

新页面或新分组应包含：

- 远程服务启用/停止。
- 访问用户名。
- 设置/修改密码。
- 监听地址/网卡。
- 监听端口。
- 允许网段。
- LAN 发现开关。
- 当前可连接地址。
- 设备指纹及复制按钮。
- 操作系统权限检查。

不提供用户列表、权限勾选、角色选择、邀请、分组或云管理页面。

### 7.3 近期连接和 LAN 发现

- 最近连接卡片显示设备名、IP/端口和用户名。
- 点击卡片使用已保存的指纹进行连接。
- LAN 发现结果可以自动填入 IP 和端口，但不应自动信任新指纹。
- 删除云地址簿、群组和服务器在线查询。

## 8. 代码改造地图

| 区域 | 主要文件/目录 | 计划动作 |
| --- | --- | --- |
| 服务启动 | `src/server.rs`, `src/core_main.rs`, `src/lib.rs` | 改为直接启动 LAN server，删除 rendezvous 生命周期 |
| 直连监听 | `src/rendezvous_mediator.rs` | 先迁出 direct server，再删除注册、打洞和 relay 实现 |
| LAN 发现 | `src/lan.rs` | 保留，将 ID 语义替换为 endpoint + fingerprint |
| 客户端连接 | `src/client.rs`, `src/client/io_loop.rs`, `src/port_forward.rs` | 收敛为单一 IP/域名直连，强制加密，复用会话循环 |
| 服务端登录 | `src/server/connection.rs`, `src/server/login_failure_check.rs` | 改为单用户 LAN 认证，认证成功即获得全功能 |

## 11. 主要风险与对策

### 11.1 “仅删除 UI”造成假 LAN-only

风险：后台仍启动 rendezvous 心跳、NAT 测试、云同步或更新网络请求。

对策：以运行时出站流量抓取、断网 E2E 和代码残留扫描共同验收，不以界面消失作为完成证据。

### 11.2 直连明文降级

风险：移除 rendezvous 后没有公钥签发，现有客户端进入未加密模式。

对策：将设备长期身份、指纹固定和 fail-closed 握手放在单用户登录之前完成，禁止任何兼容降级。

### 11.3 删除公网链路时误伤会话功能

风险：文件传输、终端、端口转发等都通过共享 `Client::start()` 或 `LoginRequest` 进入，粗暴删除会导致非桌面功能失效。

对策：先抽取单一 LAN connector，让所有 `ConnType` 切换到新入口并通过回归，再删除旧分支。

### 11.4 “无权限控制”被误实现为无平台检查

风险：移除所有 permission 逻辑可能破坏 macOS/Android/Wayland 必需权限或不支持功能的能力判定。

对策：只移除应用层用户/会话授权，保留 OS permission 和 capability 检查，并用专门测试覆盖。

### 11.5 密码哈希导致资源耗尽

风险：Argon2id 在多个恶意并发连接下消耗大量内存和 CPU。

对策：来源限速、全局验证信号量、递增退避和目标设备基准必须同时实施。

### 11.6 监听地址与 VPN 语义

风险：只允许 RFC1918 会误拦 Tailscale/IPv6/VPN；盲目监听 `0.0.0.0` 又可能暴露到公网网卡。

对策：以“明确监听网卡/地址 + 允许网段”实现 LAN-only，不使用一个无法覆盖所有 VPN 的硬编码私网判定。

## 12. 完成定义

只有同时满足以下条件，LAN-only 改造才算完成：

1. 产品主界面只提供 IP/域名、用户名和密码连接。
2. 被控端只管理一组应用访问用户名和密码。
3. 认证成功即可使用全部已保留远程功能，没有角色、权限矩阵或手动接受流程。
4. 所有连接都是直接 TCP LAN/VPN 连接，没有 ID、rendezvous、relay 或 NAT 打洞回退。
5. 应用无互联网时可完整安装、配置、发现、连接和运行会话。
6. 应用空闲或连接时不会向 RustDesk 公共/私有服务器发起请求。
7. 所有连接强制完成设备身份验证和加密，任何失败都不会降级为明文。
8. 密码不以明文或可直接重放的形式落盘，也不出现在日志中。
9. 不允许的来源网段、错误凭据、爆破尝试、指纹变化和协议降级都有自动测试。
10. 第 10.4 节的保留功能矩阵在目标发布平台全部通过。
11. 安装、升级、服务重启、被控端重启和异常断线后重连均通过验证。
12. 文档不再要求用户配置 ID server、relay server、API server 或云账号。
13. 原生桌面/移动端是 LAN-only 发布目标；Web 端不出现在发行入口中，也不以旧公网链路作为隐藏回退。

## 13. 预估工期与执行顺序

以一名熟悉 RustDesk/Rust/Flutter 的工程师全职实施估算：

| 阶段 | 预估 |
| --- | ---: |
| Phase 0：基线与依赖 | 2–3 天 |
| Phase 1：LAN server 和直连链路 | 4–6 天 |
| Phase 2：强制加密和指纹 | 5–8 天 |
| Phase 3：单用户登录 | 4–6 天 |
| Phase 4：Flutter UI | 4–7 天 |
| Phase 5：死代码和配置清理 | 3–5 天 |
| Phase 6：跨平台回归与发布 | 5–10 天 |

总计约 5–8 工程周。若第一个发布版本只将 Windows 作为被控端、Windows/macOS 作为控制端，可先将核心交付收敛到约 4–6 周；其他平台代码不删除，但不应在未验证时宣称支持。

严格执行顺序：

1. 先建立保留功能基线。
2. 再抽取 LAN 连接主链路。
3. 随后完成强制加密和单用户认证。
4. 在新主链路通过全功能回归后才删除旧公网代码。
5. 最后收敛 UI、清理残留并完成跨平台发布验收。

禁止先大规模删除 rendezvous/relay 相关类后再尝试修复文件传输、终端和端口转发；这会失去可验证的中间状态，并显著增加回归风险。
